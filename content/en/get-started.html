Quick Start

Get Started

Start using Contexa in 5 minutes. Add the starter dependency, enable AI security, and configure your infrastructure mode.

1

Add Dependency

Add the Contexa Spring Boot Starter to your project. This single dependency brings in all required modules: contexa-core, contexa-identity, contexa-iam, contexa-common, and contexa-autoconfigure.

Gradle (Kotlin DSL) 
dependencies {
    implementation("io.contexa:spring-boot-starter-contexa:0.1.0")
}
Gradle (Groovy) 
dependencies {
    implementation 'io.contexa:spring-boot-starter-contexa:0.1.0'
}
Maven 
<dependency>
    <groupId>io.contexa</groupId>
    <artifactId>spring-boot-starter-contexa</artifactId>
    <version>0.1.0</version>
</dependency>
2

Enable AI Security

Annotate your main application class with @EnableAISecurity. This activates the full Contexa AI security infrastructure, including a default PlatformConfig created through IdentityDslRegistry, which triggers the Zero Trust configurer mechanism automatically.

Java 
@SpringBootApplication
@EnableAISecurity
public class MyApplication {

    public static void main(String[] args) {
        SpringApplication.run(MyApplication.class, args);
    }
}
3

Configure

Set your infrastructure mode and AI model configuration in application.yml. Standalone mode runs with zero external dependencies. For production clusters, switch to distributed mode with Redis.

YAML 
contexa:
  infrastructure:
    mode: standalone  # or 'distributed' for Redis-backed clusters

spring:
  ai:
    chat:
      model:
        priority: ollama,anthropic,openai
    ollama:
      base-url: http://127.0.0.1:11434
      chat:
        options:
          model: qwen2.5:7b
      embedding:
        enabled: true
        model: mxbai-embed-large
    vectorstore:
      pgvector:
        table-name: vector_store
        index-type: HNSW
        distance-type: COSINE_DISTANCE
        dimensions: 1536
        initialize-schema: true

  datasource:
    url: jdbc:postgresql://localhost:5432/contexa
    username: admin
    password: ${DB_PASSWORD}
    driver-class-name: org.postgresql.Driver
Method-Level Security

Protect Methods with @Protectable

Apply AI-driven zero trust enforcement to any service method. The @Protectable annotation integrates with the XACML policy engine to evaluate access in real time.

Java 
@Service
public class OrderService {

    // Basic protection — AI evaluates every access
    @Protectable
    public Order getOrder(Long orderId) {
        return orderRepository.findById(orderId)
                .orElseThrow(() -> new OrderNotFoundException(orderId));
    }

    // Owner-based protection — only the resource owner can access
    @Protectable(ownerField = "customerId")
    public List<Order> getCustomerOrders(Long customerId) {
        return orderRepository.findByCustomerId(customerId);
    }

    // Synchronous enforcement — blocks until AI decision is made
    @Protectable(ownerField = "customerId", sync = true)
    public Order updateOrder(Long orderId, OrderRequest request) {
        Order order = orderRepository.findById(orderId)
                .orElseThrow(() -> new OrderNotFoundException(orderId));
        order.update(request);
        return orderRepository.save(order);
    }
}

@Protectable Attributes

Attribute Type Default Description
ownerField String "" The method parameter name that identifies the resource owner. When set, access is restricted to the authenticated user matching this field.
sync boolean false When true, the method blocks until the AI security analysis completes before proceeding with execution.
Authentication

Identity DSL Configuration

Configure authentication flows declaratively using IdentityDslRegistry. Compose Form Login, REST, One-Time Token, Passkey, and MFA into a single, type-safe PlatformConfig bean.

Java 
@Configuration
@EnableWebSecurity
public class PlatformSecurityConfig {

    private final CustomDynamicAuthorizationManager authorizationManager;
    private final AISessionSecurityContextRepository securityContextRepository;

    @Bean
    public PlatformConfig platformDslConfig(
            IdentityDslRegistry<HttpSecurity> registry) throws Exception {

        SafeHttpCustomizer<HttpSecurity> globalCustomizer = http -> {
            http
                .csrf(AbstractHttpConfigurer::disable)
                .authorizeHttpRequests(authReq -> authReq
                    .requestMatchers("/css/**", "/js/**", "/images/**").permitAll()
                    .anyRequest().access(authorizationManager)
                )
                .securityContext(sc ->
                    sc.securityContextRepository(securityContextRepository));
        };

        return registry
                .global(globalCustomizer)
                .mfa(mfa -> mfa
                    .primaryAuthentication(auth ->
                        auth.formLogin(form ->
                            form.defaultSuccessUrl("/dashboard")))
                    .passkey(Customizer.withDefaults()))
                .session(Customizer.withDefaults())
                .build();
    }
}

Shadow Mode recommended for production. When deploying Contexa for the first time, start in Shadow mode to observe AI security decisions without enforcing them. Once confident in AI behavior, switch to Enforce mode. This 2-stage migration path ensures zero disruption to existing traffic while validating AI behavior against your production workload.
