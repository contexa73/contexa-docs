How Contexa Works in Spring Boot

@EnableAISecurity activates a full AI-driven security pipeline on top of Spring Security. This page explains what happens under the hood — and how to customize it when the defaults aren't enough.

What @EnableAISecurity Activates

When you annotate your application with @EnableAISecurity, the following chain fires at startup:

  1. @EnableAISecurity imports AiSecurityImportSelector, which loads AiSecurityConfiguration from the auto-configure module.
  2. AiSecurityConfiguration checks for an existing PlatformConfig bean. If none exists (@ConditionalOnMissingBean), it creates a default one using IdentityDslRegistry.
  3. The presence of PlatformConfig triggers IdentitySecurityCoreAutoConfiguration, which registers the full Zero Trust filter chain — HCAD filters, access control filters, challenge filters, and event publishers.

The default PlatformConfig provides:

  • Form Login + OTT as the MFA authentication flow
  • Session management with defaults
  • AISessionSecurityContextRepository — injects AI trust scores into the security context

You may not need to customize anything. The default configuration is production-ready. The sections below are only relevant when you need to change authentication flows, authorization rules, or session behavior.

Request Lifecycle

Every HTTP request passes through these layers in order. Traditional Spring Security stops at authentication — Contexa continues with AI-driven behavioral analysis on every request.

Filter / Component What It Does
HCADFilter Extracts behavioral signals — session state, device fingerprint, IP, API access patterns, request velocity
ZeroTrustAccessControlFilter Reads the previous AI decision from the store and enforces it (ALLOW, BLOCK, CHALLENGE, ESCALATE)
ZeroTrustChallengeFilter If the decision is CHALLENGE, redirects to MFA re-authentication before proceeding
CustomDynamicAuthorizationManager Evaluates URL-level access through the XACML policy engine. Dynamic Authorization →
@Protectable Interceptor Evaluates method-level AI authorization before the method executes. @Protectable Reference →
Your Method Executes Business logic runs normally
ZeroTrustEventPublisher Publishes an async event → AI analyzes the request in the background → decision is stored and enforced on the next request

Zero latency impact. AI analysis runs asynchronously after the request completes. The current request is never blocked by the AI — the decision is enforced on the user's next request. Full Zero Trust Flow →

Customizing Authentication

Define your own PlatformConfig bean to replace the default authentication flows. Each IdentityDslRegistry method creates an independent SecurityFilterChain:

Method Creates Use Case
global()Shared HTTP configurationCSRF, authorization rules, session context
form()Form Login chainWeb UI authentication
rest()Stateless API chainREST API authentication
mfa()Multi-factor chainForm + OTT / Passkey combination
ott()One-Time Token chainMagic link authentication
passkey()WebAuthn chainPasskey authentication
session()Session managementSession state and lifecycle

REST API Only

Java 
return registry
    .global(globalHttpCustomizer)
    .rest(rest -> rest.order(10))
    .session(Customizer.withDefaults())
    .build();

Form + MFA + Passkey

Java 
return registry
    .global(globalHttpCustomizer)
    .mfa(mfa -> mfa
        .primaryAuthentication(auth -> auth
            .formLogin(form -> form.defaultSuccessUrl("/dashboard")))
        .passkey(Customizer.withDefaults())
    )
    .session(Customizer.withDefaults())
    .build();

Customizing Authorization

Use the global() customizer to define URL-level authorization rules. Static resources and public APIs should be excluded from AI evaluation to avoid unnecessary LLM calls:

Java 
SafeHttpCustomizer<HttpSecurity> globalHttpCustomizer = http -> {
    http
        .csrf(AbstractHttpConfigurer::disable)
        .authorizeHttpRequests(authReq -> authReq
            .requestMatchers("/css/**", "/js/**", "/images/**", "/favicon.ico").permitAll()
            .requestMatchers("/api/public/**").permitAll()
            .anyRequest().access(customDynamicAuthorizationManager)
        )
        .securityContext(sc -> sc.securityContextRepository(aiSessionSecurityContextRepository));
};

For method-level protection, annotate individual methods with @Protectable. The AI evaluates authorization before the method executes.

AI Session Context

AISessionSecurityContextRepository replaces Spring Security's default context repository. It augments every session with AI-computed trust scores, behavioral baselines, and threat assessments — this is what enables continuous Zero Trust evaluation across requests.

Required for Zero Trust. If you define a custom PlatformConfig, you must register AISessionSecurityContextRepository in the global() customizer via .securityContext(sc -> sc.securityContextRepository(aiSessionSecurityContextRepository)). Without it, Zero Trust will not function. State Management →

Complete Example

A production-ready configuration combining REST API, Form Login with MFA and Passkey, static resource exclusion, and AI session context:

Java 
@Configuration
@RequiredArgsConstructor
@EnableWebSecurity
public class PlatformSecurityConfig {

    private final CustomDynamicAuthorizationManager customDynamicAuthorizationManager;
    private final AISessionSecurityContextRepository aiSessionSecurityContextRepository;

    @Bean
    public PlatformConfig platformDslConfig(IdentityDslRegistry<HttpSecurity> registry) throws Exception {
        return registry
            .global(http -> http
                .csrf(AbstractHttpConfigurer::disable)
                .authorizeHttpRequests(authReq -> authReq
                    .requestMatchers("/css/**", "/js/**", "/images/**", "/favicon.ico").permitAll()
                    .requestMatchers("/api/public/**").permitAll()
                    .anyRequest().access(customDynamicAuthorizationManager))
                .securityContext(sc -> sc.securityContextRepository(aiSessionSecurityContextRepository)))
            .rest(rest -> rest.order(10))
            .mfa(mfa -> mfa
                .primaryAuthentication(auth -> auth
                    .formLogin(form -> form.defaultSuccessUrl("/dashboard")))
                .passkey(Customizer.withDefaults()))
            .session(Customizer.withDefaults())
            .build();
    }
}
