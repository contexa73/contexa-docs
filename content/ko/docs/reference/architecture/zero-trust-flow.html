AI 기반 Zero Trust 파이프라인

LLM이 모든 보안 액션을 직접 결정합니다. Spring Security가 밀리초 내에 이를 적용합니다. 규칙을 작성할 필요가 없습니다 -- 행동이 베이스라인에서 벗어나면 Contexa가 찾아냅니다.

AI Native Zero Trust가 필요한 이유

기존 보안은 인증에서 멈춥니다. 방화벽, WAF, 로그인 폼은 입구를 지킵니다 -- 하지만 세션이 수립되면 내부는 대부분 방어되지 않습니다. 탈취된 자격 증명, 침해된 세션, 그리고 밀리초 내에 인증을 통과하는 AI 에이전트 모두가 이 인증 후 사각지대를 악용합니다.

Contexa는 Spring Security와 Spring AI를 통합 AI 네이티브 Zero Trust 레이어로 융합한 최초의 플랫폼입니다. 모든 인가된 액션 -- 모든 URL 접근, 모든 @Protectable 메서드 호출 -- 은 정적 규칙이 아닌 행동 컨텍스트를 이해하는 LLM에 의해 지속적으로 평가됩니다.

기존 인증

입구만 지킵니다.

  • 로그인 시 패스워드 / SSO / MFA
  • 세션 쿠키 = 신뢰됨
  • 인증 후 모니터링 없음
  • 세션 탈취 = 완전 접근

규칙 기반 WAF

알려진 패턴을 잡습니다.

  • 시그니처 기반 탐지
  • 정적 IP / 속도 제한 규칙
  • 행동 이상 탐지 불가
  • AI 에이전트가 쉽게 우회

Contexa AI Zero Trust

컨텍스트를 이해합니다.

  • LLM이 모든 액션을 분석
  • HCAD 행동 베이스라인
  • 시그니처가 아닌 편차를 탐지
  • 5가지 적용 액션 (BLOCK/CHALLENGE/ESCALATE/ALLOW/PENDING)
  • 크로스 인스턴스 실시간 차단

ZeroTrustAction -- 5가지 결정

LLM이 액션을 직접 결정합니다. 계층형 AI 분석 파이프라인은 ZeroTrustAction을 포함하는 SecurityDecision을 출력합니다 -- 리스크 점수가 액션으로 매핑되는 것이 아닙니다. riskScore와 confidence 같은 필드는 SecurityDecision에 감사 로깅 및 학습 목적으로만 존재합니다.
액션 HTTP TTL Authority 의미
ALLOW 200 1500s - 요청이 신뢰됨. 정상 진행.
BLOCK 403 영구 ROLE_BLOCKED 위협 확인됨. 세션 무효화, 자격 증명 삭제. Redisson RTopic을 통한 크로스 인스턴스 전파.
CHALLENGE 401 1800s ROLE_MFA_REQUIRED 의심스러운 활동. MFA 챌린지 시작. 1시간 내 최대 2회 시도; 실패 시 BLOCK으로 에스컬레이션.
ESCALATE 423 300s ROLE_REVIEW_REQUIRED 인간 검토 필요. 300초 후 자동으로 BLOCK으로 승격. 승인 워크플로우를 위한 SOAR 통합.
PENDING_ANALYSIS 503 0s ROLE_PENDING_ANALYSIS AI 분석 진행 중. 결정이 완료될 때까지 응답이 래핑됨.

ZeroTrustAction은 유틸리티 메서드도 제공합니다: isBlocking()은 BLOCK과 ESCALATE에 대해 true를 반환하고; isAccessRestricted()는 BLOCK, CHALLENGE, ESCALATE에 대해 true를 반환합니다.

AI 분석 -- Layer 1 & Layer 2

ColdPathEventProcessor는 계층형 AI 전략을 오케스트레이션합니다. Layer 1은 Layer1ContextualStrategy를 통해 빠른 Tier 1 LLM으로 ~98%의 요청을 처리합니다. 신뢰도가 불충분할 경우 Layer 2는 Layer2ExpertStrategy로 에스컬레이션하여 심층 포렌식 분석을 수행합니다. 두 레이어 모두 구조화된 프롬프트 엔지니어링을 위해 SecurityPromptTemplate을, Spring AI ChatClient 실행을 위해 UnifiedLLMOrchestrator를 사용합니다.

LLM 출력은 SecurityDecision이며, ZeroTrustAction(ALLOW, BLOCK, CHALLENGE, ESCALATE)을 직접 포함합니다. riskScore와 confidence 필드는 감사 및 베이스라인 학습을 위해 기록됩니다 -- 액션 결정을 주도하지 않습니다.

Layer 1 -- 컨텍스트 분석 (Hot Path)

1
HCAD 컨텍스트 수집
세션 정보, 요청 패턴, 디바이스 핑거프린트, 행동 베이스라인 벡터로부터 SessionContext를 구축합니다
|
2
RAG 검색
유사한 과거 보안 이벤트를 위해 통합 벡터 스토어(유사도 임계값 0.5, top-k 5)를 검색합니다
|
3
세션 이력 분석
이상 패턴(비정상 시간, 낯선 경로, 급속 연속 요청)을 위해 최근 100개 액션을 분석합니다
|
4
LLM 평가 (Tier 1)
SecurityPromptTemplate으로 프롬프트를 구성하고 UnifiedLLMOrchestrator로 전송합니다. 타임아웃: LLM 10초 / 전체 15초. ZeroTrustAction이 포함된 SecurityDecision을 출력합니다.

Layer 2 -- 전문가 조사 (Cold Path)

Layer 1의 신뢰도가 설정된 임계값 이하일 때 트리거됩니다. 완전한 포렌식 분석을 위해 Tier 2 LLM을 사용합니다.

1
확장 컨텍스트
Layer 1 컨텍스트를 재사용 + 완전한 활동 프로파일로부터의 심층 행동 임베딩
|
2
확장 RAG 검색
포괄적인 위협 컨텍스트를 위한 top-k 10 확장 검색
|
3
전문가 LLM 평가 (Tier 2)
완전한 포렌식 분석. 타임아웃: 10초. Fail-closed: 타임아웃 시 BLOCK이 기본값 (ESCALATE가 아님).
|
4
SOAR 통합 (선택 사항)
ESCALATE 결정에 대해 ApprovalService가 SoarApprovalNotifier를 통해 인간 승인을 요청합니다

적용 인프라

AI가 결정을 내립니다. Spring Security의 Filter Chain과 분산 인프라가 이를 적용합니다 -- 밀리초 내에, 모든 인스턴스에 걸쳐, 심지어 응답 중에도.

BLOCK 적용

가장 공격적인 대응입니다. 크로스 인스턴스 전파와 함께 즉각적인 거부를 수행합니다.

1
SecurityDecisionEnforcementHandler
ZeroTrustActionRepository를 통해 Redis에 BLOCK을 저장합니다. blockedFlag를 설정합니다.
|
2
BlockingSignalBroadcaster
Redisson RTopic을 통해 BLOCK:{userId}를 발행합니다. 모든 인스턴스가 실시간으로 신호를 수신합니다.
|
3
BlockingDecisionRegistry
RTopic 신호를 수신합니다. O(1) 조회를 위해 로컬 ConcurrentHashMap을 업데이트합니다. 모든 요청에서 isBlocked() 체크.
|
4
BlockableResponseWrapper
BlockableServletOutputStream이 모든 write()에서 차단 상태를 확인합니다. 전송 중인 응답도 즉시 종료됩니다.
|
5
ZeroTrustAccessControlFilter
403 Forbidden을 반환합니다. 인증을 삭제하고 세션을 무효화합니다. IBlockedUserRecorder가 DB에 영속화합니다.

CHALLENGE 적용

추가 인증을 요구합니다. 분산 락으로 레이스 컨디션을 방지합니다.

1
ZeroTrustAccessControlFilter
CHALLENGE 액션을 감지합니다. ROLE_MFA_REQUIRED 권한을 부여합니다.
|
2
ChallengeMfaInitializer
분산 락을 획득합니다. MFA 상태 머신을 초기화합니다. MFA 페이지로 리다이렉트합니다.
|
3
ZeroTrustChallengeFilter
MFA 응답을 처리합니다. BlockMfaStateStore가 시도를 추적합니다 (1시간 내 최대 2회).
|
4
실패 에스컬레이션
MFA 실패 또는 타임아웃 -> 자동으로 BLOCK으로 승격.

ESCALATE 적용

인간 검토를 요구합니다. 시간 제한이 있으며 자동으로 BLOCK으로 승격됩니다.

1
ZeroTrustAccessControlFilter
423 Locked를 반환합니다. ROLE_REVIEW_REQUIRED를 부여합니다. TTL: 300초.
|
2
SOAR 워크플로우
ApprovalService.requestApproval() -> SoarApprovalNotifier가 보안 팀에 알림을 전송합니다.
|
3
관리자 오버라이드 또는 자동 승격
AdminOverrideService.approve()가 해결합니다. TTL 만료 시 (300초) -> 자동으로 BLOCK으로 승격.

크로스 인스턴스 전파

차단 결정이 모든 애플리케이션 인스턴스에 실시간으로 전파됩니다.

1
Redisson RTopic
채널: contexa:security:block-signal. 신호: BLOCK:{userId}, UNBLOCK:{userId}.
|
2
BlockedUserService
차단을 DB에 영속화합니다. BlockedUserTimeoutScheduler가 설정 가능한 타임아웃 후 자동 해제합니다.
|
3
ZeroTrustUnblockController
수동 차단 해제를 위한 관리자 엔드포인트입니다. 모든 인스턴스에 UNBLOCK 신호를 브로드캐스트합니다.

실시간 모니터링 & 학습

모든 AI 결정은 실시간으로 대시보드에 스트리밍되고, 미래 결정을 개선하기 위해 학습 루프에 피드백됩니다.

실시간 스트림

실시간 모니터링을 위한 Server-Sent Events.

  • ZeroTrustSsePublisher가 결정을 브로드캐스트
  • ZeroTrustSseController SSE 엔드포인트
  • 실시간 액션 피드: ALLOW/BLOCK/CHALLENGE/ESCALATE

관리자 대시보드

보안 운영자를 위한 시각적 제어 센터.

  • ZeroTrustPageController 웹 UI
  • 관리자 액션을 위한 ZeroTrustUnblockController
  • 결정 오버라이드를 위한 AdminOverrideService

학습 루프

모든 ALLOW 결정이 베이스라인을 개선합니다.

  • SecurityLearningService 베이스라인 업데이트
  • SecurityDecisionPostProcessor 벡터 저장
  • ZeroTrustSecurityService 세션 추적
  • ZeroTrustLogoutStrategy 로그아웃 시 정리

SpEL 통합 & 설정

SpEL 표현식에서의 Zero Trust

Spring Security의 @PreAuthorize 및 @PostAuthorize 어노테이션은 AbstractAISecurityExpressionRoot의 커스텀 표현식 메서드를 통해 Zero Trust 상태를 참조할 수 있습니다.

표현식반환 타입설명
isZeroTrustAllowed()boolean현재 사용자의 ZeroTrustAction이 ALLOW
isZeroTrustBlocked()boolean현재 사용자의 액션이 BLOCK
isZeroTrustChallenged()boolean현재 사용자의 액션이 CHALLENGE
getZeroTrustAction()ZeroTrustAction현재 ZeroTrustAction enum 값을 반환

설정 레퍼런스

프로퍼티기본값설명
security.zerotrust.enabledtrueZero Trust 파이프라인 활성화/비활성화
security.zerotrust.modeENFORCESHADOW (로그만) 또는 ENFORCE (완전 적용)
security.zerotrust.sampling.rate1.0분석할 요청 비율 (0.0-1.0)
security.zerotrust.hotpath.enabledtrueLayer 1 hot-path 분석 활성화
security.zerotrust.redis.updateIntervalSeconds30Redis 액션 업데이트 간격
contexa.infrastructure.modeSTANDALONESTANDALONE (인메모리) 또는 DISTRIBUTED (Redis+Kafka)
