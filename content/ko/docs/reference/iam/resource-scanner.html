Resource Scanner

애플리케이션 시작 시 @Protectable 메서드와 MVC 엔드포인트를 스캔하여 ManagedResource 엔티티로 등록하는 자동 리소스 탐색 시스템입니다.

개요

Contexa는 애플리케이션 내의 보호 가능한 리소스를 탐지하는 두 가지 상호 보완적인 리소스 스캐너를 제공합니다:

  • MethodResourceScanner — @Protectable이 부여된 비컨트롤러 메서드를 탐색합니다
  • MvcResourceScanner — RequestMappingHandlerMapping을 통해 Spring MVC 컨트롤러 엔드포인트를 탐색합니다

두 스캐너 모두 ResourceScanner 인터페이스를 구현하며, 초기 상태가 NEEDS_DEFINITION인 ManagedResource 엔티티를 생성합니다.

전체 파이프라인

리소스 스캔은 4단계 리소스 관리 파이프라인의 첫 번째 단계입니다: 스캔 → 정의 → 정책 → 동적 인가. 애플리케이션 시작 시 ResourceScanner 구현체가 모든 @RequestMapping 엔드포인트와 @Protectable 메서드를 탐색하여 데이터베이스에 ManagedResource 엔티티로 저장합니다. 이후 AI 엔진이 탐색된 리소스에 대해 사람이 읽기 쉬운 친숙한 이름을 일괄 생성합니다.

관리자는 Admin Dashboard의 Resource Workbench(/admin/resources)를 사용하여 탐색된 리소스를 검토하고, 권한으로 정의하며, 인가 정책을 생성합니다. 정책이 생성되면 CustomDynamicAuthorizationManager가 .anyRequest().access(customDynamicAuthorizationManager)를 통해 런타임에 정책을 적용합니다. 이 접근 방식은 정적 .requestMatchers().hasRole() 설정의 필요성을 제거합니다 — 모든 인가 결정은 관리자 인터페이스를 통해 정의된 정책에 의해 구동됩니다.

리소스 스캔 파이프라인
애플리케이션 시작 WorkbenchInitializer가 리소스 스캔을 트리거
MethodResourceScanner.scan() io.contexa.contexaiam 내의 빈 스캔 -- @Controller / @RestController 제외 -- @Protectable이 있는 public 메서드 탐색 -- ManagedResource (METHOD)로 등록
MvcResourceScanner.scan() RequestMappingHandlerMapping 조회 -- 핸들러 메서드 순회 -- URL 패턴 및 HTTP 메서드 추출 -- ManagedResource (URL)로 등록

관리자 권한 할당 status: PERMISSION_CREATED

ResourceScanner 인터페이스

Java 
public interface ResourceScanner {
    List<ManagedResource> scan();
}

MethodResourceScanner

io.contexa.contexaiam 패키지 내의 모든 Spring 빈에서 @Protectable이 부여된 public 메서드를 스캔합니다. 컨트롤러 클래스는 MvcResourceScanner에서 처리하므로 명시적으로 제외됩니다.

스캔 로직

  1. ApplicationContext의 모든 빈 정의를 순회합니다
  2. 대상 클래스를 해석합니다 (AopUtils.getTargetClass()를 통해 AOP 프록시 언래핑)
  3. io.contexa.contexaiam 패키지 외부의 클래스는 건너뜁니다
  4. @Controller 또는 @RestController가 부여된 클래스는 건너뜁니다
  5. @Protectable이 있는 각 public 메서드에 대해 ManagedResource를 생성합니다

MvcResourceScanner

RequestMappingHandlerMapping을 사용하여 Spring MVC @Controller 및 @RestController 클래스의 URL 매핑된 엔드포인트를 스캔합니다.

스캔 로직

  1. requestMappingHandlerMapping 빈을 조회합니다
  2. 등록된 모든 핸들러 메서드를 순회합니다
  3. io.contexa.contexaiam 패키지 외부의 클래스는 건너뜁니다
  4. @Controller 또는 @RestController가 부여되지 않은 클래스는 건너뜁니다
  5. PathPatternsRequestCondition에서 URL 패턴을 추출합니다
  6. HTTP 메서드를 추출합니다 (기본값은 ANY)
  7. IamAdminProperties.restDocsPath를 사용하여 API 문서 URL을 생성합니다

ManagedResource 엔티티

Java 
@Entity
@Table(name = "MANAGED_RESOURCE")
public class ManagedResource {
    private Long id;
    private String resourceIdentifier;    // URL pattern or method signature
    private ResourceType resourceType;    // URL or METHOD
    private HttpMethod httpMethod;        // GET, POST, PUT, DELETE, PATCH, ANY
    private String friendlyName;
    private String description;
    private String serviceOwner;
    private String parameterTypes;        // JSON array (METHOD only)
    private String returnType;            // Fully qualified type (METHOD only)
    private String apiDocsUrl;            // Link to API documentation
    private String sourceCodeLocation;    // Source file path (METHOD only)
    private String availableContextVariables;
    private Status status;                // NEEDS_DEFINITION, PERMISSION_CREATED,
                                          // POLICY_CONNECTED, EXCLUDED
    private Permission permission;        // OneToOne mapping
    private LocalDateTime createdAt;
    private LocalDateTime updatedAt;

    public enum ResourceType { URL, METHOD }
    public enum HttpMethod { GET, POST, PUT, DELETE, PATCH, ANY }
    public enum Status {
        NEEDS_DEFINITION,
        PERMISSION_CREATED,
        POLICY_CONNECTED,
        EXCLUDED
    }
}

Resource Workbench

Admin Dashboard의 Resource Workbench(/admin/resources)는 탐색된 리소스를 관리하기 위한 시각적 인터페이스를 제공합니다.

리소스 탐색

  • 상태 기반 필터링으로 모든 스캔된 리소스를 조회
  • 필요 시 스캔 결과를 새로 고침
  • 리소스 유형, 상태 또는 이름으로 검색 및 필터링

리소스를 권한으로 정의

NEEDS_DEFINITION 상태의 리소스에서 "Define as Permission"을 클릭하여:

  1. 대응하는 Permission 엔티티를 생성합니다
  2. 리소스 상태를 PERMISSION_CREATED로 업데이트합니다
  3. 해당 리소스를 정책 생성에 사용할 수 있도록 합니다

Workbench에서의 두 가지 경로

리소스를 권한으로 정의한 후, 두 가지 정책 생성 옵션을 사용할 수 있습니다:

경로동작사용 사례
빠른 부여"Grant via Wizard" → Policy Wizard (3단계)간단한 역할-권한 할당
고급 정책"Create Policy" → Policy Builder (5단계, from-resource 모드)복잡한 조건, AI 표현식, 다중 규칙 정책

전체 워크플로우

전체 리소스 관리 파이프라인
1단계: 스캔
MvcResourceScanner + MethodResourceScanner ManagedResource (NEEDS_DEFINITION)
2단계: 정의
Resource Workbench /admin/resources -- 권한으로 정의 (PERMISSION_CREATED)
3단계: 정책
Policy Builder / Wizard /admin/policies -- from-resource 모드 (POLICY_CONNECTED)
4단계: 적용
CustomDynamicAuthorizationManager reload() -- .anyRequest().access(dynamic)

1단계: 스캔

WorkbenchInitializer를 통해 애플리케이션 시작 시 자동으로 수행됩니다. MvcResourceScanner는 모든 @Controller 및 @RestController 클래스를 스캔합니다. MethodResourceScanner는 모든 @Protectable 메서드를 스캔합니다. AI가 일괄적으로 친숙한 이름을 생성합니다.

2단계: 검토 & 정의

Resource Workbench에서 NEEDS_DEFINITION 리소스를 검토합니다. 관련 리소스를 권한으로 정의합니다. 인가가 필요 없는 리소스(정적 자산, 공개 엔드포인트)는 제외합니다.

3단계: 정책 생성

Workbench에서 "Create Policy"를 클릭하여 from-resource 모드로 Policy Builder에 진입합니다. 호환되는 조건 템플릿이 자동으로 필터링됩니다. 규칙과 조건을 구성한 후 저장합니다.

4단계: 동적 인가

정책이 저장되면 reloadAuthorizationSystem()이 CustomDynamicAuthorizationManager.reload()를 트리거합니다. 정책은 즉시 적용됩니다. 보안 설정에 .anyRequest().access(customDynamicAuthorizationManager)가 있으므로 정적 인가 규칙이 필요 없습니다.

Resource Workbench: 운영 가이드

Resource Workbench(/admin/resources)는 탐색된 모든 리소스를 관리하기 위한 중앙 인터페이스입니다. 이 섹션에서는 상세한 운영 워크플로우를 다룹니다.

리소스 필터링

Workbench는 리소스를 빠르게 찾기 위한 다양한 필터링 옵션을 제공합니다:

필터옵션사용 사례
상태별NEEDS_DEFINITION, PERMISSION_CREATED, POLICY_CONNECTED, EXCLUDED특정 생명주기 단계의 리소스 찾기
서비스 소유자별컨트롤러 또는 서비스 클래스명특정 모듈에 속한 리소스 필터링
키워드별자유 텍스트 검색리소스 식별자, 친숙한 이름 또는 설명으로 검색
리소스 유형별URL 또는 METHOD엔드포인트 유형 vs. 메서드 수준 리소스 필터링

리소스 정의

리소스를 정의하면 다음 단계가 순서대로 진행됩니다:

  1. NEEDS_DEFINITION 리소스에서 "Define as Permission"을 클릭합니다
  2. 권한에 대한 친숙한 이름을 입력합니다 (또는 AI가 제안한 이름을 수락합니다)
  3. Permission 엔티티가 자동으로 생성되어 리소스에 연결됩니다
  4. 리소스 상태가 PERMISSION_CREATED로 변경됩니다
  5. 이제 리소스가 정책 할당에 사용할 수 있는 상태가 됩니다

Workbench에서 정책 설정

리소스를 정의한 후 두 가지 정책 생성 경로를 제공하는 모달이 나타납니다:

빠른 부여 (Wizard)

"Grant via Wizard"를 클릭하여 3단계 Policy Wizard를 엽니다.

  • 리소스 권한이 미리 선택됩니다
  • 대상 역할을 선택하고 확인합니다
  • 간단한 ALLOW 정책을 생성합니다
VS

고급 (Policy Builder)

"Create Policy"를 클릭하여 5단계 Policy Builder를 엽니다.

  • 대상이 미리 채워집니다 (from-resource 모드)
  • 전체 조건 템플릿 선택 가능
  • 복잡한 규칙과 AI 조건을 지원합니다

리소스 제외 및 복원

인가가 필요 없는 리소스(공개 엔드포인트, 정적 자산, 헬스 체크)는 제외할 수 있습니다:

  • 제외: NEEDS_DEFINITION 리소스에서 "Exclude"를 클릭합니다. 상태가 EXCLUDED로 변경되고, 리소스가 인가 관리에서 제거됩니다.
  • 복원: 제외된 리소스는 "Restore"를 클릭하여 복원할 수 있습니다. 상태가 NEEDS_DEFINITION으로 되돌아가고, 리소스가 관리 생명주기에 다시 진입합니다.

제외해야 할 경우: 공개 API 엔드포인트, 정적 자산 경로, 헬스 체크 엔드포인트(/actuator/health), 문서 경로.

복원해야 할 경우: 이전에 공개였지만 이제 인가가 필요한 엔드포인트, 또는 실수로 제외된 리소스.

AI 자동 이름 지정

스캐너에 의해 리소스가 탐색되면, 기술적 식별자(URL 패턴 또는 메서드 시그니처)는 사용자 친화적이지 않습니다. AI 자동 이름 지정 시스템은 자동으로 사람이 읽기 쉬운 이름과 설명을 생성합니다.

작동 방식

ResourceRegistryServiceImpl은 스캔이 완료된 후 AI ResourceNaming 서비스를 호출합니다. AI는 각 기술적 식별자를 분석하여 다음을 생성합니다:

  • 친숙한 이름: 간결하고 비즈니스 지향적인 이름 (예: PUT /api/users/{id}에 대해 "Update User Profile")
  • 설명: 리소스가 수행하는 작업에 대한 더 긴 설명

일괄 처리

AI API 사용을 최적화하기 위해, 자동 이름 지정은 배치당 10개 항목으로 리소스를 처리합니다. 이 접근 방식은:

  • AI API 호출 횟수를 줄입니다
  • 같은 배치 내의 관련 리소스에 대한 문맥 인식을 제공합니다
  • 개별 배치를 재시도하여 실패를 우아하게 처리합니다
