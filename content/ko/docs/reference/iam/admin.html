Admin Dashboard

보안 개요, 사용자/역할/그룹/권한 관리, 블랙리스트 관리, 정책 편집을 위한 시각적 Authorization Studio를 제공하는 웹 기반 관리 콘솔입니다.

메뉴 구조

Admin Dashboard는 기능을 논리적 그룹으로 구성합니다:

메뉴 그룹페이지목적
Dashboard /admin/dashboard 보안 메트릭 및 통계 개요
Identity Users, Groups, Roles, Permissions 사용자 및 접근 엔티티 관리
Authorization Policies, Policy Builder, Studio 정책 생성 및 분석
Resources Resource Workbench 리소스 탐색, 권한 정의
Workflows Policy Wizard, Granting Wizard 가이드형 다단계 워크플로우
Security Blacklist, Role Hierarchies 보안 운영

컨트롤러

모든 관리 컨트롤러는 /admin 기본 경로 하에서 제공되며 Thymeleaf 템플릿 뷰를 반환합니다. @Controller 어노테이션을 사용하는 표준 Spring MVC 규칙을 따릅니다.

컨트롤러기본 경로목적
DashboardController /admin 집계된 통계를 포함한 보안 개요 대시보드
AuthorizationStudioController /admin/studio 정책 탐색, 시뮬레이션 및 부여 워크플로우를 위한 시각적 인가 스튜디오
UserManagementController /admin/users 그룹 할당이 포함된 사용자 CRUD
RoleController /admin/roles 권한 할당이 포함된 역할 CRUD
PermissionController /admin/permissions 관리 리소스 연결이 포함된 권한 CRUD
GroupController /admin/groups 역할 할당이 포함된 그룹 CRUD
BlacklistController /admin/blacklist 해제 워크플로우가 포함된 차단 사용자 관리

DashboardService의 집계된 데이터를 사용하여 메인 보안 개요 페이지를 제공합니다.

엔드포인트메서드설명
/admin, /admin/, /admin/dashboardGET보안 통계를 포함한 대시보드 뷰를 렌더링합니다

Authorization Studio

접근: /admin/studio

Authorization Studio는 인가 시스템을 위한 고급 분석 및 시각화 도구를 제공합니다. 동적 상호작용을 위해 페이지 엔드포인트와 JSON API 엔드포인트를 모두 제공합니다.

Subject Explorer

사용자, 그룹 및 역할을 탐색하고 검색합니다. 할당된 권한, 그룹 멤버십 및 역할 계층을 조회합니다.

접근 경로 분석

특정 사용자가 그룹, 역할, 권한 및 정책의 체인을 통해 특정 리소스에 대한 접근 권한을 얻는 과정을 추적합니다.

유효 권한

다음을 고려하여 사용자가 가진 전체 권한 집합을 조회합니다:

  • 직접 역할 할당
  • 그룹 멤버십
  • 역할 계층 상속

정책 시뮬레이션

"what-if" 시나리오를 테스트합니다: 역할, 권한 또는 그룹 멤버십의 추가 또는 제거를 시뮬레이션하고 접근 결정이 어떻게 변경되는지 확인합니다.

부여 워크플로우

필요한 역할/권한 할당을 생성하여 Studio 인터페이스에서 직접 접근 권한을 부여합니다.

BlacklistController

Zero Trust 엔진에 의해 감지된 차단된 사용자를 관리합니다. 상태별 필터링과 관리자의 해제 워크플로우를 지원합니다.

엔드포인트메서드설명
/admin/blacklistGET차단된 사용자를 목록으로 조회합니다. 필터 파라미터: all, blocked, unblock_requested, resolved, timeout_responded
/admin/blacklist/{id}GET차단 상세를 표시합니다
/admin/blacklist/{id}/resolvePOST조치와 사유로 차단을 해제합니다. 현재 관리자가 기록됩니다.
/admin/blacklist/{id}/deletePOST차단 기록을 삭제합니다

BlockedUserStatus 값

상태설명
BLOCKEDZero Trust 엔진에 의해 현재 차단된 사용자
UNBLOCK_REQUESTED사용자가 차단 해제를 요청한 상태
RESOLVED관리자가 차단을 해제한 상태
TIMEOUT_RESPONDED타임아웃으로 차단이 만료되고 사용자가 응답한 상태

Resource Workbench

접근: /admin/resources (Enterprise 에디션)

Resource Workbench는 Resource Scanner에 의해 탐색된 리소스를 관리하기 위한 시각적 인터페이스를 제공합니다.

리소스 탐색

  • 모든 @RequestMapping 엔드포인트와 @Protectable 메서드가 시작 시 자동으로 스캔됩니다
  • 리소스는 NEEDS_DEFINITION 상태로 표시됩니다
  • AI가 생성한 친숙한 이름이 리소스 식별에 도움을 줍니다

리소스 정의

  • Define as Permission을 클릭하여 리소스에 대한 Permission 엔티티를 생성합니다
  • 상태가 PERMISSION_CREATED로 변경됩니다

리소스에서 정책 생성

두 가지 경로를 사용할 수 있습니다:

  • 빠른 부여: Grant via Wizard를 클릭하여 간단한 역할-권한 할당을 위한 3단계 Policy Wizard를 엽니다
  • 고급 정책: Create Policy를 클릭하여 호환 조건이 자동 필터링된 from-resource 모드의 5단계 Policy Builder를 엽니다

전체 워크플로우는 Resource Scanner 페이지를 참조하세요.

워크플로우 Wizard

Policy Wizard

접근: /admin/policy-wizard

역할에 권한을 할당하기 위한 3단계 가이드 워크플로우입니다:

단계동작
1단계대상 역할을 선택합니다
2단계할당할 권한을 선택합니다
3단계검토하고 확인합니다

Granting Wizard

접근: /admin/granting-wizard

사용자/그룹 멤버십 관리를 위한 가이드 워크플로우입니다:

  • 사용자를 그룹에 할당합니다
  • 역할을 그룹에 할당합니다
  • 실시간 시뮬레이션: 적용 전에 멤버십 변경의 영향을 미리 봅니다
  • 변경 전후의 유효 권한을 보여줍니다

시작하기 시나리오

애플리케이션 배포부터 완전한 동적 인가까지의 전체 안내입니다:

단계동작상세
1 애플리케이션 배포 Resource Scanner가 자동으로 모든 엔드포인트를 탐색합니다
2 Resource Workbench 열기 탐색된 리소스를 검토합니다 (NEEDS_DEFINITION 상태)
3 권한 정의 인가가 필요한 리소스에 대해 Define as Permission을 클릭합니다
4 정책 생성 Workbench에서 Policy Builder를 사용하여 적절한 조건의 정책을 생성합니다
5 검증 핫 리로드를 통해 정책이 즉시 적용됩니다. 다른 사용자 역할로 접근을 테스트합니다.
6 모니터링 Authorization Studio를 사용하여 접근 경로를 분석하고 변경을 시뮬레이션합니다

이 워크플로우가 완료되면, 애플리케이션의 인가는 정적 보안 설정 없이 Admin Dashboard를 통해 완전히 관리됩니다.

사용자 관리 가이드

사용자 관리 화면(/admin/users)은 완전한 사용자 생명주기 관리를 제공합니다.

사용자 생성

  1. /admin/users로 이동하여 "New User"를 클릭합니다
  2. 사용자 프로필을 입력합니다: 사용자명, 이메일, 이름
  3. 초기 비밀번호를 설정합니다 (또는 외부 ID 제공자용으로 구성합니다)
  4. 선택적으로 사용자를 하나 이상의 그룹에 할당합니다
  5. "Save"를 클릭하여 사용자를 생성합니다

사용자 프로필 편집

목록에서 사용자를 클릭하면 상세 페이지가 열립니다. 여기서 다음을 수행할 수 있습니다:

  • 프로필 정보 업데이트 (이름, 이메일)
  • 그룹 멤버십 변경
  • 유효 권한 조회 (그룹과 역할을 통해 상속된 권한)
  • 사용자의 인증 이력 및 현재 세션 상태 확인

그룹 및 역할 할당

사용자는 다음 체인을 통해 권한을 얻습니다: 사용자 → 그룹 → 역할 → 권한. 사용자를 그룹에 할당하려면:

  1. 사용자 상세 페이지를 엽니다
  2. "Groups" 섹션에서 사용 가능한 목록에서 그룹을 선택합니다
  3. 변경을 저장합니다 -- 사용자는 할당된 그룹의 모든 역할과 권한을 즉시 상속받습니다

MFA 상태 관리

각 사용자의 다중 인증(Multi-Factor Authentication) 상태를 조회하고 관리합니다:

  • Enabled: 사용자가 MFA를 구성하고 활성화한 상태
  • Disabled: MFA가 구성되지 않은 상태
  • Reset: 관리자가 인증기 기기에 대한 접근을 잃은 사용자의 MFA 설정을 초기화할 수 있습니다

역할 관리 가이드

역할 관리 화면(/admin/roles)은 역할 생성 및 권한 할당을 처리합니다.

역할 생성

  1. /admin/roles로 이동하여 "Register Role"을 클릭합니다
  2. 역할 이름을 입력합니다 (없는 경우 자동으로 ROLE_ 접두사가 추가됩니다)
  3. 역할의 목적을 설명하는 설명을 추가합니다
  4. 사용 가능한 목록에서 이 역할에 할당할 권한을 선택합니다
  5. "Save"를 클릭하여 역할을 생성합니다

역할에 권한 할당

역할의 상세 페이지를 열어 권한 할당을 관리합니다:

  • 현재 할당된 모든 권한을 조회합니다
  • 권한 선택기를 사용하여 권한을 추가하거나 제거합니다
  • 권한이 변경되면 RolePermissionsChangedEvent가 발생하고 PolicySynchronizationService가 대응하는 정책을 자동으로 생성/업데이트합니다

역할 계층 구성

역할은 부모-자식 계층으로 구성할 수 있습니다. 부모 역할은 자식 역할의 모든 권한을 자동으로 상속받습니다. 자세한 내용은 역할 계층 구성을 참조하세요.

그룹 관리 가이드

그룹 관리 화면(/admin/groups)은 사용자와 역할을 논리적 그룹으로 구성합니다.

그룹 생성

  1. /admin/groups로 이동하여 "Register Group"을 클릭합니다
  2. 그룹 이름과 설명을 입력합니다
  3. 이 그룹에 할당할 역할을 선택합니다
  4. "Save"를 클릭하여 그룹을 생성합니다

그룹에 역할 할당

그룹의 상세 페이지를 열어 역할 할당을 관리합니다. 그룹의 모든 사용자는 할당된 역할의 권한을 자동으로 상속받습니다.

그룹 멤버십 관리

사용자를 그룹에 추가하거나 그룹에서 제거하는 방법은 두 가지입니다:

  • 그룹 페이지에서: 모든 멤버를 조회하고 사용자를 추가/제거합니다
  • 사용자 페이지에서: 사용자 프로필에서 그룹에 할당합니다
  • Granting Wizard를 통해: 영향 미리보기와 함께 대량 할당을 위한 가이드 워크플로우(/admin/granting-wizard)를 사용합니다

권한 관리 가이드

권한 관리 화면(/admin/permissions)은 개별 권한 엔티티를 처리합니다.

권한 생성

  1. /admin/permissions로 이동하여 "Register Permission"을 클릭합니다
  2. 권한 상세를 입력합니다:
필드설명예시
Name고유 권한 식별자USER_READ
Description사람이 읽기 쉬운 설명Read user profile information
Target Type이 권한이 적용되는 도메인USER, DOCUMENT, ORDER
Action Type이 권한이 부여하는 동작READ, WRITE, DELETE

권한과 관리 리소스 연결

권한은 Resource Scanner에 의해 탐색된 ManagedResource 엔티티에 연결할 수 있습니다. 이 연결은 일반적으로 Resource Workbench에서 "Define as Permission"을 사용할 때 자동으로 생성되지만, 권한 상세 페이지에서 수동으로 구성할 수도 있습니다.

권한 생명주기

권한은 다음과 같은 생명주기를 따릅니다:

  1. 생성: Permission 엔티티가 생성됩니다 (수동 또는 Resource Workbench를 통해)
  2. 역할 할당: 권한이 하나 이상의 역할에 할당됩니다
  3. 정책 연결: 정책이 SpEL 조건을 통해 권한을 참조합니다
  4. 적용: CustomDynamicAuthorizationManager가 런타임에 권한을 평가합니다

블랙리스트 관리

블랙리스트 관리 화면(/admin/blacklist)은 Zero Trust 엔진에 의해 차단된 사용자를 처리합니다.

Zero Trust 차단 작동 방식

Zero Trust 엔진은 사용자 행동과 AI 리스크 분석을 지속적으로 모니터링합니다. 의심스러운 활동이 감지되면, 엔진이 자동으로 사용자를 차단할 수 있습니다. 차단 결정은 다음에 의해 구동됩니다:

  • AI 리스크 점수: 설정된 임계값을 초과
  • 행동 이상: 비정상적인 접근 패턴, 불가능한 이동, 크레덴셜 스터핑
  • 정책 위반: 비인가 리소스에 대한 반복적인 접근 시도
  • SecurityDecisionEnforcementHandler: ZeroTrustAction.BLOCK 결정을 처리

차단된 사용자 조회

블랙리스트 화면은 상태별 필터링으로 모든 차단된 사용자를 표시합니다:

  • BLOCKED: 현재 차단됨, 관리자 해제가 필요함
  • UNBLOCK_REQUESTED: 사용자가 차단 해제 요청을 제출함
  • RESOLVED: 관리자가 차단을 해제함
  • TIMEOUT_RESPONDED: 타임아웃으로 차단이 만료됨

차단 해제 워크플로우

  1. /admin/blacklist로 이동하여 차단된 사용자를 찾습니다
  2. 사용자 항목을 클릭하여 차단 상세를 확인합니다 (사유, 타임스탬프, 리스크 점수)
  3. 차단을 트리거한 AI 분석을 검토합니다
  4. "Resolve"를 클릭하고 조치를 선택합니다:
  • Unblock: 차단을 제거하고 접근을 복원합니다
  • Keep blocked: 문서화와 함께 차단을 확인합니다
  • Escalate: 추가 보안 조사를 위해 플래그합니다

해제한 관리자의 신원은 감사 목적으로 기록됩니다.

역할 계층 구성

역할 계층은 역할 간의 권한 상속을 가능하게 합니다. 부모 역할은 자식 역할의 모든 권한을 자동으로 포함합니다.

부모-자식 관계 설정

  1. /admin/roles로 이동하여 부모가 될 역할을 선택합니다
  2. 역할 상세 페이지에서 부모 역할 필드를 구성합니다
  3. 자식 역할의 권한이 부모에게 자동으로 상속됩니다

계층이 권한 상속에 미치는 영향

역할 계층 권한 상속
ROLE_ADMIN 자식 역할의 모든 권한을 상속 + 자체 권한
ROLE_MANAGER ROLE_USER 권한을 상속 + 자체 권한
ROLE_AUDITOR 읽기 전용 접근 권한
ROLE_USER 기본 수준 권한

이 예에서 ROLE_ADMIN은 ROLE_MANAGER, ROLE_AUDITOR, ROLE_USER의 권한을 상속받습니다. 계층은 Spring Security의 RoleHierarchy 구현에 의해 해석되며, Authorization Studio의 유효 권한 계산과 CustomDynamicAuthorizationManager의 런타임 평가 모두에서 고려됩니다.

기본 정책 설정

기본 정책 설정은 요청이 구성된 정책과 일치하지 않을 때의 동작을 제어합니다.

미매칭 요청 허용 또는 거부

CustomDynamicAuthorizationManager가 요청을 평가할 때 일치하는 정책이 없으면, 시스템은 기본 정책 설정으로 대체합니다:

설정동작권장 용도
DENY (기본값)미매칭 요청이 거부됩니다운영 환경 -- deny-by-default가 가장 안전한 자세입니다
ALLOW미매칭 요청이 허용됩니다개발 환경 또는 모든 리소스에 정책이 아직 없는 초기 마이그레이션 시

Admin UI를 통한 관리

기본 정책 설정은 데이터베이스에 저장되며 애플리케이션을 재시작하지 않고 Admin Dashboard를 통해 변경할 수 있습니다:

  1. Dashboard 설정 섹션으로 이동합니다
  2. "Default Authorization Policy" 설정을 찾습니다
  3. ALLOW와 DENY 사이를 토글합니다
  4. 핫 리로드를 통해 변경이 즉시 적용됩니다 (무중단)

보안 권장 사항

  • 운영: 항상 DENY를 기본값으로 사용하세요. 접근 가능해야 하는 모든 리소스에 대해 명시적으로 ALLOW 정책을 생성하세요.
  • 스테이징: 운영 동작과 일치시키고 배포 전에 누락된 정책을 찾기 위해 DENY를 사용하세요.
  • 개발: 초기 설정 시 ALLOW를 사용하고, 정책이 구성되면 DENY로 전환하세요. Shadow Mode를 사용하여 요청을 차단하지 않고 정책 적용을 테스트하세요.
