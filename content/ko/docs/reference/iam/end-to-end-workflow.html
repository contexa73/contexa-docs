End-to-End 동적 인가 워크플로우

하드코딩된 규칙 없이 완전한 정책 기반 인가 시스템을 구축합니다. 자동 리소스 탐색부터 런타임 적용까지 — Contexa는 Spring Security의 정적 인가를 완전히 동적인 데이터베이스 기반 접근 방식으로 대체합니다.

왜 혁신적인가

전통적인 Spring Security는 개발자가 Java 설정에 인가 규칙을 직접 하드코딩하도록 강제합니다. 새로운 엔드포인트, 역할 변경, 권한 수정마다 코드 변경, 재빌드, 재배포가 필요합니다. Contexa는 이 전체 패턴을 제거합니다.

전통적 방식 vs. Contexa 접근 방식

전통적인 Spring Security

Java 
http.authorizeHttpRequests(auth -> auth
    .requestMatchers("/api/admin/**")
        .hasRole("ADMIN")
    .requestMatchers("/api/users/**")
        .hasAuthority("READ_USERS")
    .requestMatchers("/api/reports/**")
        .hasAnyRole("ADMIN", "ANALYST")
    .requestMatchers("/api/orders/**")
        .hasAuthority("MANAGE_ORDERS")
    .requestMatchers("/api/settings/**")
        .hasRole("SUPER_ADMIN")
    .requestMatchers("/api/audit/**")
        .hasAuthority("VIEW_AUDIT_LOG")
    // ... 20+ more lines
    .anyRequest().authenticated()
);

모든 변경에 코드 수정, 재빌드, 재배포가 필요합니다. 소스 코드를 읽지 않고는 누가 무엇에 접근할 수 있는지 알 수 없습니다.

VS

Contexa 동적 인가

Java 
http.authorizeHttpRequests(auth -> auth
    .requestMatchers("/css/**", "/js/**")
        .permitAll()
    .anyRequest()
        .access(customDynamicAuthorizationManager)
);

한 줄로 모든 정적 규칙을 대체합니다. 정책은 Admin Dashboard를 통해 관리되고, 데이터베이스에 저장되며, 런타임에 적용됩니다. 무중단. 재배포 불필요.

주요 이점

기능 전통적 방식 Contexa
정책 변경 코드 변경 + 재빌드 + 재배포 Admin Dashboard 편집, 즉시 적용
새 엔드포인트 보호 수동 requestMatchers 항목 ResourceScanner에 의한 자동 탐색
정책 생성 개발자가 Java 코드 작성 Admin UI wizard 또는 AI 지원 빌더
감사 추적 Git 이력만 CentralAuditFacade를 통한 완전한 감사 로그
AI 통합 불가능 AI 리스크 평가, 조건 추천
세분화 수준 URL 패턴 + 역할 URL, 메서드, 소유권, 시간, IP, AI 조건
가시성 소스 코드 읽기 모든 이해관계자를 위한 중앙 집중식 대시보드

전체 워크플로우 개요

동적 인가 시스템은 5단계 파이프라인으로 운영됩니다. 각 단계는 자동화되어 있으며 다음 단계로 이어져, 코드에서 적용까지 매끄러운 흐름을 만듭니다.

ResourceScanner
시작 시 모든 엔드포인트와 @Protectable 메서드를 자동 탐색
registers
Resource Workbench
탐색된 리소스를 탐색, 이름 지정, 구성하는 Admin UI
configures
Policy Builder / Wizard
수동 또는 AI 지원으로 정책 생성
persists
DB Policy Storage
정책, 조건, 대상이 데이터베이스에 저장
loads
Dynamic Authorization Manager
모든 요청에 대해 런타임에 정책 평가

다음 섹션에서는 이 파이프라인이 실제로 작동하는 네 가지 실제 시나리오를 안내합니다.

시나리오 1 — 새 API 엔드포인트 보호

새로운 REST 컨트롤러를 작성했으며, 보안 설정을 건드리지 않고 동적으로 보호하려고 합니다.

1
@RequestMapping이 있는 컨트롤러 작성

평소처럼 엔드포인트를 생성합니다. 보안 어노테이션은 필요 없습니다.

Java 
@RestController
@RequestMapping("/api/v1/invoices")
public class InvoiceController {

    @GetMapping
    public List<InvoiceDto> listInvoices() {
        return invoiceService.findAll();
    }

    @PostMapping
    public InvoiceDto createInvoice(@RequestBody CreateInvoiceRequest req) {
        return invoiceService.create(req);
    }

    @DeleteMapping("/{id}")
    public void deleteInvoice(@PathVariable Long id) {
        invoiceService.delete(id);
    }
}
2
시작 시 ResourceScanner 자동 감지

MvcResourceScanner는 애플리케이션 시작 시 자동으로 실행됩니다. 모든 컨트롤러에서 @RequestMapping, @GetMapping, @PostMapping, @PutMapping, @DeleteMapping, @PatchMapping 어노테이션을 스캔합니다.

세 개의 새로운 리소스가 데이터베이스에 등록됩니다:

리소스 ID HTTP 메서드 URL 패턴 타입
GET:/api/v1/invoices GET /api/v1/invoices URL
POST:/api/v1/invoices POST /api/v1/invoices URL
DELETE:/api/v1/invoices/{id} DELETE /api/v1/invoices/{id} URL

수동 등록이 필요 없습니다. 스캐너가 모든 것을 자동으로 탐색합니다.

3
Resource Workbench — 구성 및 이름 지정

Admin Dashboard를 열고 Resource Workbench로 이동합니다. 세 개의 새 엔드포인트가 리소스 목록에 나타납니다.

각 리소스에 대해 다음을 수행할 수 있습니다:

  • 사람이 읽기 쉬운 비즈니스 이름 할당 (예: "List Invoices", "Create Invoice", "Delete Invoice")
  • 리소스를 리소스 그룹으로 그룹화 (예: "Invoice Management")
  • 리소스에 매칭되는 Permission이 자동으로 생성됩니다 (예: INVOICES_READ, INVOICES_CREATE, INVOICES_DELETE)
4
정책 생성 — 빠른 부여 또는 고급 빌더

인가 정책을 생성하는 두 가지 옵션이 있습니다:

옵션 A: 빠른 부여 (Policy Wizard)

가장 빠른 경로입니다. 역할을 선택하고, 권한을 할당하면 완료됩니다.

  1. 리소스를 선택합니다 (예: GET:/api/v1/invoices)
  2. 대상 역할을 선택합니다: ADMIN, ACCOUNTANT
  3. 권한을 할당합니다: INVOICES_READ
  4. 효과: ALLOW
  5. "Create Policy"를 클릭합니다

이것은 hasAnyRole('ADMIN', 'ACCOUNTANT')에 해당하는 간단한 역할 기반 정책을 생성합니다 — 단, 코드가 아닌 데이터베이스에 저장됩니다.

옵션 B: 고급 Policy Builder

조건이 포함된 복잡한 인가 로직을 위한 옵션입니다.

Policy Definition (Conceptual) 
Policy: "Invoice Deletion - Restricted"
  Target:   DELETE:/api/v1/invoices/{id}
  Effect:   ALLOW
  Roles:    ADMIN
  Conditions:
    - Time:  weekdays 09:00-18:00 only
    - IP:    internal network (10.0.0.0/8)
    - AI:    #ai.riskScore(request) < 0.7

Policy Builder는 이를 런타임에 SpEL 표현식을 사용하여 평가되는 XACML 호환 조건으로 변환합니다.

5
런타임 적용 — 자동 및 즉시

정책은 데이터베이스에 저장됩니다. CustomDynamicAuthorizationManager가 PolicyRetrievalPoint(캐싱 포함)를 통해 정책을 로드하고 런타임 매처 목록을 구성합니다.

DELETE /api/v1/invoices/42 요청이 들어오면:

  1. Spring Security가 CustomDynamicAuthorizationManager에 위임합니다
  2. 관리자가 요청 URL을 "Invoice Deletion - Restricted" 정책에 매칭합니다
  3. 모든 조건이 평가됩니다: 역할 검사, 시간 창, IP 범위, AI 리스크 점수
  4. 모든 조건이 통과하면 AuthorizationDecision(true)이 반환됩니다
  5. CentralAuditFacade가 비동기적으로 결정을 로깅합니다

결과: 엔드포인트가 동적으로 보호됩니다. 코드 변경 없음. 재배포 없음. 정책은 Admin Dashboard를 통해 언제든지 업데이트할 수 있습니다.

시나리오 2 — 비즈니스 메서드 보호

URL 수준 인가를 넘어, Contexa는 @Protectable 어노테이션을 사용한 메서드 수준 보호를 지원합니다. 이를 통해 소유권 의미론을 가진 개별 서비스 메서드에 대한 세밀한 접근 제어가 가능합니다.

1
@Protectable 어노테이션 적용
Java 
@Service
public class DocumentService {

    @Protectable(ownerField = "createdBy")
    public Document getDocument(Long documentId) {
        return documentRepository.findById(documentId)
            .orElseThrow(() -> new NotFoundException("Document not found"));
    }

    @Protectable(ownerField = "createdBy")
    public void updateDocument(Long documentId, UpdateDocumentRequest req) {
        Document doc = documentRepository.findById(documentId)
            .orElseThrow(() -> new NotFoundException("Document not found"));
        doc.setTitle(req.getTitle());
        doc.setContent(req.getContent());
        documentRepository.save(doc);
    }
}

ownerField 파라미터는 반환된 엔티티에서 리소스 소유자를 식별하는 필드를 Contexa에 알려줍니다. 이를 통해 소유권 기반 인가가 가능합니다: 소유자는 항상 자신의 리소스에 접근할 수 있습니다.

2
시작 시 MethodResourceScanner 감지

MethodResourceScanner가 모든 빈에서 @Protectable 어노테이션을 스캔하고 METHOD 타입 리소스로 등록합니다:

리소스 ID 클래스 메서드 타입
DocumentService.getDocument DocumentService getDocument METHOD
DocumentService.updateDocument DocumentService updateDocument METHOD
3
Resource Workbench에 METHOD 리소스 표시

Admin Dashboard의 Resource Workbench에서 METHOD 타입 리소스가 URL 리소스와 함께 표시됩니다. 시각적으로 구분되며 동일한 방식으로 관리할 수 있습니다 — 비즈니스 이름 할당, 그룹화, 권한 생성이 가능합니다.

4
소유권 + AI 조건이 포함된 Policy Builder

"from-resource" 모드에서 Policy Builder를 사용하여 소유권 검사와 추가 조건을 결합하는 정책을 생성합니다:

Policy Definition (Conceptual) 
Policy: "Document Update - Owner or Admin"
  Target:   DocumentService.updateDocument
  Effect:   ALLOW
  Rules:
    Rule 1 (Owner Access):
      Condition: #ownership.isOwner(authentication, returnObject)
      Effect:    ALLOW
    Rule 2 (Admin Override):
      Condition: hasRole('ADMIN')
      Effect:    ALLOW
    Rule 3 (AI Risk Gate):
      Condition: #ai.riskScore(request) > 0.8
      Effect:    DENY (overrides all)

이 정책은 문서 소유자 또는 ADMIN이 문서를 업데이트할 수 있도록 허용하지만, AI 리스크 평가가 임계값을 초과하면 동작을 완전히 차단합니다.

5
동적 메서드 수준 인가 활성화

정책이 이제 메서드 수준에서 적용됩니다. updateDocument()가 호출되면, Contexa의 AOP 인터셉터가 메서드 실행 전에 정책을 평가합니다. 소유권은 authentication.getName()과 엔티티의 createdBy 필드를 비교하여 해석됩니다.

시나리오 3 — AI 지원 정책 생성

Contexa는 정책 생성 워크플로우에 AI를 직접 통합합니다. AI Policy Builder는 리소스를 분석하고, 조건을 추천하며, 리스크를 평가하고, 활성화 전에 승인 워크플로우를 거치는 완전한 정책을 생성할 수 있습니다.

AI 기반 리스크 평가

민감한 리소스에 대한 정책을 생성할 때, AI 엔진은 다음을 분석합니다:

  • 리소스 민감도 — 접근하는 데이터 유형 기반 (PII, 금융, 건강 기록)
  • 과거 접근 패턴 — 누가 이 리소스에 접근했으며 얼마나 자주인지
  • 위협 환경 — 현재 보안 상황 및 알려진 공격 벡터
  • 규제 요구사항 — 적용 가능한 규정 준수 프레임워크 (GDPR, HIPAA, PCI-DSS)

조건 자동 추천

분석을 기반으로 AI가 정책에 대한 조건을 추천합니다:

AI-Generated Recommendations 
Resource: DELETE:/api/v1/customers/{id}
AI Analysis: HIGH SENSITIVITY (PII data, destructive operation)

Recommended Conditions:
  1. Role restriction:     ADMIN only
  2. Time restriction:     Business hours (M-F 09:00-18:00)
  3. IP restriction:       Internal network only
  4. MFA requirement:      Step-up authentication required
  5. AI risk gate:         #ai.riskScore(request) < 0.5
  6. Rate limit:           Max 10 deletions per hour per user

Confidence: 92%
Rationale: Destructive operation on PII data requires
           maximum protection with multiple condition layers.

승인 워크플로우

AI가 생성한 정책은 즉시 활성화되지 않습니다. 엄격한 승인 워크플로우를 따릅니다:

AI가 정책 생성
Status: DRAFT
submit
검토 대기
Status: PENDING
approve
활성 정책
Status: APPROVED

APPROVED 상태의 정책만 CustomDynamicAuthorizationManager에 의해 로드됩니다. 보안 관리자는 AI가 생성한 정책이 적용되기 전에 검토합니다.

정책에서의 AI 표현식

승인되면, AI 기반 조건은 SpEL 표현식을 사용하여 런타임에 평가됩니다:

Java (SpEL) 
// Allow if AI determines the request is safe
#ai.isAllowed(request, authentication)

// Require step-up authentication if AI detects anomalies
#ai.needsChallenge(request, authentication)

// Numeric risk scoring for threshold-based decisions
#ai.riskScore(request) < 0.7

// Combined with traditional conditions
hasRole('ADMIN') AND #ai.riskScore(request) < 0.5

시나리오 4 — 정적에서 동적으로 마이그레이션

이 시나리오는 20줄 이상의 하드코딩된 requestMatchers가 있는 기존 애플리케이션을 Contexa의 동적 인가로 전환하는 과정을 안내합니다.

이전: 정적 보안 설정

Java 
@Bean
SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http.authorizeHttpRequests(auth -> auth
        .requestMatchers("/css/**", "/js/**", "/images/**").permitAll()
        .requestMatchers("/api/public/**").permitAll()
        .requestMatchers("/api/admin/users/**").hasRole("ADMIN")
        .requestMatchers("/api/admin/settings/**").hasRole("SUPER_ADMIN")
        .requestMatchers("/api/admin/audit/**").hasAuthority("VIEW_AUDIT")
        .requestMatchers("/api/reports/financial/**").hasAnyRole("ADMIN", "FINANCE")
        .requestMatchers("/api/reports/analytics/**").hasAnyRole("ADMIN", "ANALYST")
        .requestMatchers("/api/orders/**").hasAuthority("MANAGE_ORDERS")
        .requestMatchers("/api/customers/**").hasAuthority("MANAGE_CUSTOMERS")
        .requestMatchers("/api/products/**").hasAuthority("MANAGE_PRODUCTS")
        .requestMatchers("/api/inventory/**").hasAnyRole("ADMIN", "WAREHOUSE")
        .requestMatchers(HttpMethod.DELETE, "/api/**").hasRole("ADMIN")
        .requestMatchers(HttpMethod.PUT, "/api/users/*/role").hasRole("SUPER_ADMIN")
        // ... more rules
        .anyRequest().authenticated()
    );
    return http.build();
}
1
ResourceScanner 활성화 (자동 구성)

Contexa IAM 의존성을 추가합니다. MvcResourceScanner는 자동 구성되어 시작 시 실행됩니다. 기존의 모든 엔드포인트가 자동으로 탐색되어 리소스로 등록됩니다.

YAML 
contexa:
  iam:
    resource-scanner:
      enabled: true          # default: true
      scan-on-startup: true  # default: true
2
Workbench에서 탐색된 리소스 검토

Admin Dashboard의 Resource Workbench를 엽니다. 기존의 모든 엔드포인트가 나열됩니다. 검토하고, 비즈니스 이름을 할당하며, 그룹으로 구성합니다. 이를 통해 애플리케이션의 모든 보호 대상 영역에 대한 완전한 인벤토리를 얻습니다.

3
권한 정의 및 정책 생성

각 리소스(또는 리소스 그룹)에 대해 Policy Wizard를 사용하여 동등한 정책을 생성합니다. 기존 정적 규칙을 동적 정책으로 매핑합니다:

정적 규칙 동적 정책
.hasRole("ADMIN") Policy: roles=[ADMIN], effect=ALLOW
.hasAuthority("VIEW_AUDIT") Policy: permissions=[VIEW_AUDIT], effect=ALLOW
.hasAnyRole("ADMIN","FINANCE") Policy: roles=[ADMIN,FINANCE], effect=ALLOW

마이그레이션이 완료되면, 정적 설정에서는 불가능했던 조건으로 이러한 정책을 즉시 강화할 수 있습니다: 시간 기반 접근, IP 제한, AI 리스크 평가 등.

4
정적 설정 교체

전체 정적 보안 설정을 하나의 동적 라인으로 교체합니다:

Java 
@Bean
SecurityFilterChain securityFilterChain(HttpSecurity http,
        CustomDynamicAuthorizationManager customDynamicAuthorizationManager)
        throws Exception {
    http.authorizeHttpRequests(auth -> auth
        .requestMatchers("/css/**", "/js/**", "/images/**").permitAll()
        .requestMatchers("/api/public/**").permitAll()
        .anyRequest().access(customDynamicAuthorizationManager)
    );
    return http.build();
}

정적 매처는 진정한 공개 리소스(자산, 공개 API)에만 남아 있습니다. 나머지는 모두 동적 인가 관리자에게 위임됩니다.

5
기본 정책 구성

명시적 정책이 없는 리소스에 대한 기본 동작을 설정합니다. 자세한 내용은 아래의 기본 정책 관리 섹션을 참조하세요.

기본 정책 관리

요청이 명시적 정책이 없는 리소스와 매칭될 때, 시스템에는 대체 동작이 필요합니다. 이것이 기본 정책입니다.

현재 기본 동작

기본적으로 Contexa의 CustomDynamicAuthorizationManager는 매칭되지 않는 리소스에 대해 new AuthorizationDecision(true)을 반환합니다. 이는 리소스에 대한 정책이 없으면 접근이 허용됨을 의미합니다 (사용자가 인증된 경우).

Java 
// Default behavior: allow authenticated users if no policy matches
if (matchingPolicies.isEmpty()) {
    return new AuthorizationDecision(true);
}

기본 거부 구성

더 엄격한 보안 자세(운영 환경에 권장)를 위해, 명시적으로 인가된 리소스만 접근 가능하도록 기본 거부를 구성합니다:

YAML 
contexa:
  iam:
    authorization:
      default-decision: DENY  # ALLOW or DENY
      log-unmatched: true     # log requests with no matching policy

Admin Dashboard를 통한 관리

기본 정책은 애플리케이션 재시작 없이 Admin Dashboard를 통해 변경할 수도 있습니다. Settings > Authorization > Default Policy로 이동하여 ALLOW와 DENY 사이를 토글합니다. 데이터베이스 기반 설정을 통해 변경이 즉시 적용됩니다 — 무중단.

핵심 컴포넌트

CustomDynamicAuthorizationManager

AuthorizationManager<RequestAuthorizationContext>를 구현합니다. 초기화 시 PolicyRetrievalPoint를 통해 데이터베이스에서 모든 활성 정책을 로드하고 RequestMatcherEntry 객체 목록을 구성합니다. 각 항목은 URL 패턴(HTTP 메서드 포함)을 정책의 조건을 평가하는 ExpressionAuthorizationManager에 매핑합니다.

PolicyRetrievalPoint (PRP)

정책의 데이터 접근 계층입니다. 데이터베이스에서 정책을 로드하고, 성능을 위해 캐시하며, Admin Dashboard를 통해 정책이 수정될 때 핫 리로드를 지원합니다. 캐시 무효화는 이벤트 기반입니다 — 정책이 생성, 업데이트 또는 삭제되면 PRP가 알림을 받고 캐시를 새로 고칩니다.

PolicyTranslator

XACML 스타일의 정책 조건을 SpEL(Spring Expression Language) 표현식 트리로 변환합니다. Admin Dashboard의 사람이 읽기 쉬운 정책 정의와 Spring Security의 런타임 표현식 평가 엔진 사이의 간격을 연결합니다.

CentralAuditFacade

모든 인가 결정이 비동기적으로 로깅됩니다. 감사 로그는 요청 상세, 매칭된 정책, 평가된 조건, 결정 결과, 사용자 신원, 타임스탬프를 캡처합니다. 이를 통해 규정 준수 및 포렌식 분석을 위한 완전한 감사 추적을 제공합니다.

  • Policy Management — 정책 생명주기, CRUD 작업, AI 보강
  • @Protectable — 메서드 수준 리소스 보호 어노테이션
  • XACML 통합 — 정책 모델 및 평가 엔진

  • Admin Dashboard — 웹 기반 관리 콘솔
