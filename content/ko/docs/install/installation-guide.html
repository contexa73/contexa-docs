설치 가이드

Contexa AI Native Zero Trust 플랫폼의 전체 설정 안내로, 사전 요구 사항, 의존성 구성, Standalone 및 Distributed 배포 모드를 다룹니다.

사전 요구 사항

Contexa를 설치하기 전에 환경이 다음 요구 사항을 충족하는지 확인하세요:

요구 사항 버전 용도
Java 21+ 런타임 (Gradle 툴체인으로 구성)
Spring Boot 3.x 애플리케이션 프레임워크
PostgreSQL 15+ with pgvector Identity 저장소 및 벡터 임베딩
Ollama 최신 버전 AI 보안 분석을 위한 로컬 LLM 추론

Distributed 모드 전용 — Distributed 모드를 사용할 계획이라면 Redis(비동기 보안 컨텍스트용)와 Kafka(이벤트 스트리밍용)도 필요합니다.

Starter 의존성 추가

빌드 파일에 spring-boot-starter-contexa를 추가하세요. 이 단일 의존성은 모든 Contexa 모듈과 자동 구성을 전이적으로 포함합니다.

Gradle (Kotlin DSL) 
dependencies {
    implementation("io.contexa:spring-boot-starter-contexa:0.1.0")
}
Gradle (Groovy) 
dependencies {
    implementation 'io.contexa:spring-boot-starter-contexa:0.1.0'
}
Maven 
<dependency>
    <groupId>io.contexa</groupId>
    <artifactId>spring-boot-starter-contexa</artifactId>
    <version>0.1.0</version>
</dependency>

AI 보안 활성화

메인 애플리케이션 클래스에 @EnableAISecurity를 추가하여 Zero Trust 구성 메커니즘을 활성화하고 기본 PlatformConfig를 등록하세요.

Java 
import io.contexa.contexacommon.annotation.EnableAISecurity;

@SpringBootApplication
@EnableAISecurity
public class MyApplication {

    public static void main(String[] args) {
        SpringApplication.run(MyApplication.class, args);
    }
}

내부 동작 원리 — @EnableAISecurity는 AiSecurityImportSelector를 임포트하여 AI 보안 인프라를 조건부로 등록합니다. 커스텀 PlatformConfig 빈이 없으면 IdentityDslRegistry를 통해 기본 설정이 자동으로 생성됩니다.

데이터베이스 설정

Contexa는 Identity 저장소와 AI 보안 분석에 사용되는 벡터 임베딩을 위해 pgvector 확장이 설치된 PostgreSQL이 필요합니다.

SQL 
-- Create the database
CREATE DATABASE identity;

-- Connect to the database and enable required extensions
\c identity

CREATE EXTENSION IF NOT EXISTS vector;
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";

자동 스키마 생성 — 필요한 모든 테이블은 ddl-auto가 update로 설정되면 Hibernate에 의해 자동으로 생성됩니다. 추가 마이그레이션 스크립트를 실행할 필요가 없습니다.

Ollama 설정

Contexa는 로컬 LLM 추론을 위해 Ollama를 사용합니다. Ollama를 설치하고 채팅 및 임베딩에 필요한 모델을 내려받으세요.

Shell 
# Install Ollama from https://ollama.ai

# Pull the chat model
ollama pull qwen2.5:7b

# Pull the embedding model
ollama pull mxbai-embed-large

리소스 요구 사항 — qwen2.5:7b 모델은 약 4.7 GB의 디스크 공간과 추론을 위해 최소 8 GB의 RAM이 필요합니다. 모델을 내려받기 전에 시스템에 충분한 리소스가 있는지 확인하세요.

모드 A: Standalone (메모리 모드)

Standalone 모드는 기본 배포 모드입니다. InMemoryAsyncSecurityContextProvider를 사용하며 PostgreSQL과 Ollama 외에 추가 인프라가 필요하지 않습니다.

적합한 환경 — 개발 환경, 단일 인스턴스 배포, 빠른 시작에 적합합니다. Redis나 Kafka가 필요하지 않습니다.

다음 구성으로 application.yml을 생성하거나 업데이트하세요:

YAML 
contexa:
  infrastructure:
    mode: standalone

spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/identity
    username: your_username
    password: your_password
    driver-class-name: org.postgresql.Driver

  jpa:
    database: POSTGRESQL
    database-platform: org.hibernate.dialect.PostgreSQLDialect
    hibernate:
      ddl-auto: update

  ai:
    ollama:
      base-url: http://127.0.0.1:11434
      embedding:
        enabled: true
        model: mxbai-embed-large
      chat:
        options:
          model: qwen2.5:7b
          keep-alive: "24h"

    vectorstore:
      pgvector:
        table-name: vector_store
        index-type: HNSW
        distance-type: COSINE_DISTANCE
        dimensions: 1536
        initialize-schema: true

  auth:
    token-transport-type: header_cookie
    oauth2-csrf: false

모드 B: Distributed

Distributed 모드는 여러 애플리케이션 인스턴스 간에 공유 보안 컨텍스트를 위해 RedisAsyncSecurityContextProvider를 사용하고, 이벤트 스트리밍을 위해 Kafka를 사용합니다.

추가 인프라 필요 — Distributed 모드는 기본 사전 요구 사항 외에 Redis(기본값: localhost:6379)와 Kafka(기본값: localhost:9092)가 필요합니다.

Standalone 설정 위에 다음 구성을 추가하세요:

YAML 
contexa:
  infrastructure:
    mode: distributed
    redis:
      enabled: true
    kafka:
      enabled: true

spring:
  data:
    redis:
      host: localhost
      port: 6379
      timeout: 200000
      lettuce:
        pool:
          max-active: 8
          max-idle: 8

  cache:
    type: redis
    redis:
      time-to-live: 3600000

  kafka:
    bootstrap-servers: localhost:9092

Standalone 구성과 병합 — Distributed 구성은 추가적입니다. Standalone 구성의 datasource, JPA, AI, auth 설정을 모두 유지하고 위에 표시된 Redis/Kafka 섹션을 추가하세요.

검증

애플리케이션을 시작한 후 다음 방법 중 하나를 사용하여 Contexa가 올바르게 구성되었는지 확인하세요.

방법 1: 디버그 로깅

--debug 플래그로 애플리케이션을 실행하고 자동 구성 보고서의 positive matches 섹션에서 CoreInfrastructureAutoConfiguration을 찾으세요.

Shell 
./gradlew bootRun --args='--debug'

# Look for in the output:
# Positive matches:
#   CoreInfrastructureAutoConfiguration matched

방법 2: Actuator Health 엔드포인트

Spring Boot Actuator가 활성화되어 있다면 health 엔드포인트를 확인하여 애플리케이션이 정상적으로 실행되고 있는지 확인하세요.

Shell 
curl http://localhost:8080/actuator/health

문제 해결

설치 중 발생하는 일반적인 문제와 해결 방법입니다.

문제 원인 해결 방법
PostgreSQL 연결 거부 데이터베이스가 실행되지 않거나 자격 증명이 올바르지 않음 datasource URL, 사용자명/비밀번호를 확인하고 클라이언트 인증 설정을 위해 pg_hba.conf를 점검하세요.
Ollama 연결 거부 Ollama 서비스가 실행되지 않음 ollama serve로 Ollama를 시작하고 포트 11434에서 수신 대기 중인지 확인하세요.
pgvector 확장을 찾을 수 없음 PostgreSQL에 확장이 설치되지 않음 데이터베이스에서 CREATE EXTENSION IF NOT EXISTS vector;를 실행하세요. pgvector 패키지를 먼저 설치해야 할 수 있습니다.
포트 충돌 다른 프로세스가 필요한 포트를 사용 중 기본 포트: PostgreSQL 5432, Redis 6379, Kafka 9092, Ollama 11434. netstat 또는 lsof로 충돌을 확인하세요.
Redis 연결 거부 (Distributed 모드) Redis가 실행되지 않거나 호스트/포트가 올바르지 않음 Redis를 시작하고 구성된 호스트와 포트에서 접근 가능한지 확인하세요.

추가 도움이 필요하신가요? — 사용 가능한 모든 속성의 전체 목록은 구성 참조를 확인하거나, 커뮤니티 지원을 위해 GitHub Discussions를 방문하세요.
